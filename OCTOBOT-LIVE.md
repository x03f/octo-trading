# Живой бумажный контур OctoBot (23.07.2026)

## 🔓 Главное: как чинится «кастомная стратегия не работает в OctoBot»
Причина, по которой это не работало весь проект: **мало положить файлы в `tentacles/` и включить
тентакл в профиле** — его надо УСТАНОВИТЬ через менеджер, иначе он не попадает в
`user/reference_tentacles_config/tentacles_config.json` и OctoBot его просто не грузит:
```bash
cd <install> && OctoBot tentacles -sti /путь/к/пакету "Evaluator/TA" -d <install>
```
Пакет = папка: `__init__.py` (импорт класса), `<module>.py`, `metadata.json` (список классов),
`config/<Class>.json`, `resources/`. **Реестр у каждого инстанса свой** → ставить в каждый.

Ещё три грабли, на которых я посидел:
1. **`required_candles_count`** в `config/<Class>.json` обязателен — иначе свечей не хватает и
   `evaluate()` не вызывается (OctoBot берёт МАКСИМУМ по активным тентаклам).
2. **Профиль адресуется по `profile.id`**, а не по имени папки. Если `config.json → profile` не
   совпадает с `profile.json → profile.id`, OctoBot молча откатывается на `default`.
3. **Профиль `default` удалять нельзя** — стартап падает с `KeyError: 'default'`.
4. В режиме **бэктеста** кастомный TA-эвалюатор вызывается лишь раз (на старте) — прокрутка его не
   триггерит. Не чинил: бэктест честнее в своём движке, OctoBot нужен для ЖИВОГО режима.

## ⚖️ Про плечо — позиция пересмотрена по фактам
Раньше здесь было написано «поставил рабочее 3x, на х10 ликвидация приходит на обычном движении».
После починки сайзинга это утверждение стало неточным и заменено.

Плечо само по себе не убивает — убивает РАЗМЕР позиции. При `0.5%` на ×100 одна позиция ≈ $4.5k
на счёт $10k: маржа ~$45, до ликвидации нужно движение ~1%, и потеря ограничена этой маржой.
Совокупный условный объём по 10 монетам ≈ $45k = **эффективное плечо ×4.5, а не ×100**.
То есть «×100» у нас теперь про эффективность маржи, а не про риск.

Что действительно опасно на высоком плече — комиссии: круговая на ×100 съедает ~4% маржи,
поэтому 5m-скальп на lev100 требует, чтобы движение стабильно перекрывало косты. Это и проверяем.

# 🔴 Аудит логов 23.07.2026: почему 0 сделок на всех 7 инстансах

Проверены логи всех доменов. Инстансы живые (NRestarts=0, эвалюаторы считают), но **ни одной сделки
за 4.5 часа ни на одном**. Найдены три независимые проблемы, две из них — блокирующие.

## ⛔ ПРИЧИНА №1 (корневая): сигнал не доходит до торгового режима
**Расхождение путей в матрице оценок.** Проверено прямым прогоном на классах OctoBot:

| Кто | Что делает | Путь в матрице |
|---|---|---|
| `BlankStrategyEvaluator` | ПУБЛИКУЕТ ноту (`strategy_completed(..., time_frame=tf)`) | `['BTC', 'BTC/USDT', '1h']` |
| `DailyTradingMode.set_final_eval` | ЧИТАЕТ ноту (`get_tentacles_value_nodes` **без** `time_frame`) | `['BTC', 'BTC/USDT']` |

Режим читает узел-КОНТЕЙНЕР уровня символа, у которого `node_value = None`.
Далее `check_valid_eval_note(None, float, float)` → **False** → `strategies_analysis_note_counter`
остаётся 0 → `create_state()` не вызывается → состояние не меняется → ордера не создаются.

Подтверждающие факты из логов (все 7 инстансов):
- строк `new state:` — **0**;
- строк `Order creation inputs` — **0**;
- при этом строк `EVALUATOR_TYPE = STRATEGIES` — сотни (реле работает, его просто никто не слушает).

**Вывод: связка `BlankStrategyEvaluator` + `DailyTradingMode` нерабочая в принципе.** Blank
ретранслирует ноту вместе с таймфреймом — это годится для скриптовых/DSL-режимов, читающих узлы
уровня TF, но не для `DailyTradingMode`. Штатные стратегии (`SimpleStrategyEvaluator`) агрегируют
несколько TF и публикуют на уровне СИМВОЛА — потому у них связка работает.

**Починка:** свой мини-тентакл-реле, публикующий БЕЗ таймфрейма:
`self.eval_note = eval_note; await self.strategy_completed(cryptocurrency, symbol)`.
Двадцать строк. Blank в профилях заменить на него.

## ✅ ПРОБЛЕМА №2 (ПОЧИНЕНО): плечо лежало не в том файле
Симптом был: `linear perpetual isolated **x1**` во всех трёх фьючерсных инстансах.

**OctoBot плечо УМЕЕТ, и в симуляторе тоже** — вопреки первой версии этого разбора.
Механизм лежит в БАЗОВОМ классе `AbstractTradingModeProducer._apply_exchange_side_config()`,
то есть достаётся ЛЮБОМУ торговому режиму, включая `DailyTradingMode`:
```python
if self.trading_mode.is_updating_exchange_settings(context):   # → True, никто не переопределяет
    await script_keywords.set_leverage(context, await script_keywords.user_select_leverage(context))
```
`user_select_leverage(ctx, def_val=1, name="leverage")` — это **user-input торгового режима**.
`Trader.set_leverage()` в симуляции работает: биржевой вызов пропускается (`if not self.simulate`),
но `contract.set_current_leverage()` выполняется всегда.

**Настоящая ошибка была моя и чисто адресная:** провижининг клал `"leverage": N` в корень
`profile.json`, откуда его не читает никто. Класть надо в конфиг ТОРГОВОГО РЕЖИМА:
`specific_config/DailyTradingMode.json`. Ключ там же, где `max_currency_percent` и `use_stop_orders`.

Проверено 23.07 в 18:49: после переноса ключа и рестарта в логах
`Switching ETH/USDT:USDT leverage from 1 to 100` — по 10 пар на каждом инстансе, **0 отказов**,
позиции теперь `linear perpetual isolated x3 / x10 / x100`.
Даже x100 приняли все 10 мажоров — `contract.check_leverage_update()` не отклонил ничего.

⚠️ **Провижининг `provision_instance.sh` всё ещё пишет ключ в старое место** — при создании нового
фьючерсного инстанса плечо снова окажется x1. Чинить в скрипте.

- **`trader.enabled=false` не мешает.** Декоратор `enabled_trader_only` пропускает симулятор;

- **Ротация логов:** у `squeeze-wide` и `squeeze-futures` живой процесс пишет в `OctoBot.log.1`

# ✅ Починка 23.07.2026 вечер: конвейер пробит насквозь

## Блокер №1 закрыт: свой тентакл-реле
Создан `tentacle_src/relay_strategy_evaluator/` — `RelayStrategyEvaluator`. Отличие от штатного
`BlankStrategyEvaluator` ровно одно и оно решающее:
```python
await self.strategy_completed(cryptocurrency, symbol)      # БЕЗ time_frame
```
Нота ложится по пути `[крипта, символ]` — туда, где её ищет `DailyTradingMode`.
Ставится как `"Evaluator/Strategies"`, в профиле включается вместо Blank, и в
`DailyTradingMode.json` → `required_strategies: ["RelayStrategyEvaluator"]`.

## Диагностический тентакл (новый инструмент, беречь)
`tentacle_src/smoketest_evaluator/` — `SmokeTestEvaluator`, пила `-1 → 0 → +1 → 0` независимо от цены.
**Зачем:** боевая стратегия может молчать сутками (штиль → нет пробоя), и «работает, но сигнала нет»
неотличимо от «сломано». Именно на этом мы потеряли день. Теперь проходимость конвейера
проверяется за 2 минуты на любом свежем инстансе.
⚠️ В боевом профиле активным оставаться НЕ ДОЛЖЕН.

Результат дымового теста на lev100 (19:06): `NOTE=-1` → `new state: VERY_LONG` ×10 →
`Order creation inputs` ×10 → **16 сделок исполнено**. До этого за 4.5 часа было 0.

## ⛔ НОВАЯ НАХОДКА: `max_currency_percent` на фьючерсах НЕ РАБОТАЕТ
Дымовой тест снёс счёт с $10 000 до **$0.24** за одну минуту. Причина найдена в исходнике
`daily_trading.py`, там же авторская пометка о незавершённости:
```python
def _get_max_amount_from_max_ratio(self, max_ratio, quantity, currency, default_ratio):
    # TODO ratios in futures trading
    if self.MAX_CURRENCY_RATIO is None or max_ratio == ONE or self.exchange_manager.is_future:
        return quantity * default_ratio        # ← на фьючерсах лимит просто игнорируется
```
А `QUANTITY_MARKET_MAX_PERCENT = 1` зашит константой. Итог: рыночный ордер на фьючерсах берёт
**весь плечевой запас** по каждому символу отдельно.

Факт из лога: на счёте $10 000 открыто **$1 857 819 условного объёма** за 16 сделок,
комиссий $928.91 (9.3% счёта за минуту). Одна сделка по ADA — $593 394 на счёт в $10k.

**Следствия:**
1. Заявленный «лимит 20% капитала на монету» на фьючерсах был фикцией. На споте он работает.
2. Плечо ×100 с `DailyTradingMode` смертельно не из-за рынка, а из-за сайзинга.
3. **Решение:** ключи `buy_order_amount` / `sell_order_amount` в `DailyTradingMode.json`.
   Синтаксис DSL: `"5%"` — процент от баланса счёта, `"a%"` — от доступного, `"q"` — в котируемой
   валюте. Это ЕДИНСТВЕННЫЙ работающий способ ограничить размер позиции на фьючерсах.

⚠️ Оговорка о честности: пила дымового тентакла переворачивает позицию каждую свечу, поэтому
комиссионный ожог преувеличен против реальной стратегии. Но **размер позиции** от сигнала не
зависит — $593k на счёт $10k сломано при любой стратегии.

## Почему инстансов много, а не один (архитектурный факт, не наш выбор)
OctoBot жёстко запрещает больше одного торгового режима на процесс —
`octobot_trading/modes/mode_config.py`:
```python
if len(trading_modes) > 1:
    raise errors.ConfigTradingError("More than one activated trading mode found ... please activate only one.")
```
Один процесс = один профиль = один торговый режим = один портфель = одна веб-панель на своём порту.
Отсюда: N стратегий → N процессов → N портов → N поддоменов. Плюс отдельные $10 000 на инстанс
дают чистую атрибуцию P&L: в общем портфеле нельзя понять, какая стратегия заработала.

## 📊 Первый живой результат ×100 (23.07.2026 19:24) — ликвидация как штатный исход

lev100 открыл шорты BTC и SOL по сигналу S8 и **обе позиции были ликвидированы за минуты**.
Это не поломка, а измерение — ровно то, ради чего контур ×100 и поднимался.

| | значение |
|---|---|
| счёт | $10 000 → $9 904.68 (−0.95%) |
| условный объём 2 позиций | $9 078 |
| маржа при ×100 | $90.78 |
| комиссии | $4.54 |
| **потеря минус комиссии** | **$90.78 — ровно вся маржа, до цента** |

**Что это доказывает:** защита размером позиции работает в точности как задумано. Ликвидация
съедает ТОЛЬКО маржу позиции ($45 на монету), а не счёт. Сравнить с прошлым разом, когда
без `buy_order_amount` те же $10k открыли $1.86 млн и умерли за минуту.

**Что это говорит про ×100 по существу:** одна ликвидация стоит **0.48% счёта**, до нуля хватит
~209 подряд. На 5m-пробое с ликвидацией при движении ~1% такие события будут частыми, а не
редкими. Значит вопрос к ×100 не «выживет ли счёт сегодня», а «перекрывает ли редкий выигрыш
поток мелких ликвидаций». Это и есть предмет измерения; ответа пока нет — нужна статистика.

**Побочный эффект починки:** 900+ предупреждений `evaluation on outdated data` исчезли полностью.
Реле не передаёт `time_frame`, а `_log_on_invalid_eval_not_time` при пустом ТФ выходит сразу.
Логи стали читаемыми: 0 ошибок и 0 предупреждений на шести инстансах из семи.
