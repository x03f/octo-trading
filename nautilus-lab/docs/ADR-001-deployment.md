# ADR-001: способ развёртывания — нативно (systemd + venv), без Docker

Статус: **принято** (24.07.2026). Решение обязательно к исполнению во всём проекте.

## Решение
**Полностью нативное развёртывание: systemd-юниты + Python venv + нативный PostgreSQL 17.**
Docker НЕ используется ни полностью, ни частично.

## Почему (сравнение трёх вариантов)
Сервер уже держит ~10 проектов (shnurok, shnalytics, folio, nmh, graphify, octobot) — ВСЕ нативно
через systemd+venv, PostgreSQL 17 и MariaDB нативные, Caddy+Authelia нативные. Docker на сервере
отсутствует. Вводить его ради одного проекта = чужеродный слой на весь сервер.

| Критерий | Docker (полный) | Гибрид | **Нативно (выбран)** |
|---|---|---|---|
| Стабильность live-торговли | лишний слой сети/IO | БД в контейнере — риск | прямой доступ, минимум слоёв |
| Задержки/производительность | overhead сети/namespace | смешанный | нативный, лучший |
| Восстановление после ребута | нужен daemon + compose | два механизма | один — systemd, как весь сервер |
| Изоляция research/live | namespaces | частично | systemd unit + отдельные venv/пользователи |
| Хранение PG/озеро | volumes, сложнее бэкап | смешанный | как у остальных проектов сервера |
| Секреты | env в контейнере | смешанный | systemd EnvironmentFile 600 + Authelia |
| Эксплуатация на ЭТОМ сервере | новый стек для админа | двойной | единообразно с остальными 9 проектами |
| Риск: обновление research ломает live | ниже | средний | изолируем отдельными unit — достаточно |

Решающие факторы: (1) непрерывность live-торговли — меньше слоёв = меньше точек отказа;
(2) единообразие с сервером — админ уже знает systemd, не знает Docker здесь;
(3) grub/загрузчик на сервере хрупкий (память проекта) — не добавляем инфраструктурных рисков.

Изоляция «research не должен ронять live» достигается РАЗНЫМИ systemd-юнитами и venv, а не
контейнерами: тяжёлый бэктест — отдельный процесс/юнит с CPU/IO-лимитами (systemd slices),
live-runtime — отдельный юнит с Restart=always.

## Стек (нативный)
- **Торговое ядро:** Python 3.13 + NautilusTrader 1.230 (venv `/opt/octobot/nautilus-venv`).
- **Backend API:** FastAPI + Uvicorn (systemd-юнит), Pydantic-схемы.
- **Реестр исследований/метаданные:** PostgreSQL 17 (нативный, БД `nautilus_lab`).
- **Озеро/аналитика:** Parquet + DuckDB + Polars (файлы, без сервера БД).
- **Задачи-раннер:** нативный — таблица очереди в PG + systemd-таймер/воркер (без Celery/Redis;
  Redis отсутствует, вводить ради одной очереди на одном хосте избыточно).
- **Дашборд:** одностраничное SPA, отдаётся FastAPI (см. ADR-002).
- **Автозапуск/восстановление:** systemd `Restart=always` для live/API, `enable` для автостарта,
  `Persistent=true` таймеры для research/снимков.

## Отклонено с обоснованием (не тащим ради галочки из ТЗ)
- **TimescaleDB** — озеро уже в parquet+DuckDB (эффективнее для бэктеста, чем строки в PG);
  operational-ряды (equity live) невелики, обычной PG-таблицы достаточно.
- **Redis** — на сервере нет; runtime-state Nautilus держит в своём кэше/PG; очередь — таблица PG.
- **Celery/Dramatiq** — один хост, десятки экспериментов в день, не тысячи; systemd-воркер проще
  и надёжнее, без брокера.
Эти компоненты можно добавить, если нагрузка вырастет; сейчас они = сложность без выгоды.

## Основные команды эксплуатации (единый способ)
```bash
ntlab up        # поднять все сервисы (API, data, paper-runner)
ntlab down      # остановить
ntlab status    # состояние сервисов + health
ntlab logs <svc>
ntlab backtest <config>
ntlab restore   # восстановление из последнего снимка
```
(обёртка над systemd, см. `scripts/ntlab`)
